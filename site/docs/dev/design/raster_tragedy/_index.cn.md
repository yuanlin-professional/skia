这是 [The Raster Tragedy at Low-Resolution Revisited](http://rastertragedy.com) 在 Skia 中的应用延伸。光栅化悲剧（The Raster Tragedy）描述了字体光栅化的许多问题，特别强调了通过适当的微调（hinting）来克服这些问题。由于并非所有字体都有良好的微调，有时也不需要微调，因此还有一些额外的技巧可以应用。通常，人们希望对为特定设备排版的纯信息性文本进行微调，但不对作为艺术的文本进行微调。当然，如果微调本身就是艺术的一部分（如 Shift_JIS 艺术），则另当别论。



伽马技巧（The Gamma Hack）
--------------

首先，应该了解传递函数（transfer function）（"伽马"就是其中一个例子）。可以在 [What Every Coder Should Know About
Gamma](https://blog.johnnovak.net/2016/09/21/what-every-coder-should-know-about-gamma/) 获得一个很好的入门介绍。

在 Skia 中，所有颜色源都被转换到目标色彩空间（color space），混合在目标色彩空间中通过应用线性混合函数完成。Skia 不会转换到线性空间、应用线性混合，然后再转换回编码空间。如果目标色彩空间没有线性编码，这将导致"不正确"的混合。其思想是 Skia 基本上有两类用户。第一类是已经使用非线性编码配合线性混合函数的现有系统。虽然混合不正确，但这些用户通常不希望因为期望值而改变任何东西。第二类是希望一切都正确完成的用户，他们愿意为线性编码的目标付出代价，在此目标中线性混合函数是正确的。

对于二值（bi-level）字形渲染，一个像素要么被覆盖要么不被覆盖，因此不存在覆盖率混合问题。

对于常规的全像素部分覆盖（抗锯齿）字形渲染，用户可能需要也可能不需要正确的线性混合。在大多数非线性编码中，使用线性混合函数往往会使白底黑字看起来略微加粗，利用像素网格作为一种对比度和光学尺寸增强。对于黑底白字则相反，通常使这类字形的覆盖率略显不足。然而，这与常见的光晕效应（blooming）问题相抗衡——在许多显示器上，亮色在暗色背景上往往比暗色在亮色背景上看起来更粗。（黑色不够纯黑也有影响。）如果像素足够小，并且有适当的光学尺寸调整（optical sizing），也许还有抗锯齿丢弃控制（drop out control）（后两者通过适当的字体选择或 'opsz' 手动实现、自动实现或通过微调实现），那么正确的线性混合往往效果很好。否则，白底黑字文本在小尺寸下往往会（正确地）变得非常纤细。因此，这里字形蒙版的正确混合应由 Skia 的用户来决定。如果他们足够专业，已经解决了这些问题，则可能希望使用字形的线性混合以获得最佳效果。否则，由于期望值的原因，字形应保持原来的样子。

对于亚像素部分覆盖（亚像素抗锯齿）字形蒙版，在线性编码中进行线性混合或多或少是必需的，以避免色彩边缘效应（color fringing）。亚像素的强度被直接利用，因此需要仔细控制。亚像素往往可以缓解无完全覆盖的问题（但如果以显示器的某个原色 (primary) 绘制文本时仍有问题）。由于线性缩小时字形仍然会看起来偏淡，因此仍然需要光学尺寸调整。

那么，如果亚像素抗锯齿字形蒙版（有时包括全像素抗锯齿字形蒙版）需要正确的位块传输（blit），它们如何与非线性编码的目标一起使用？

一种可能的解决方案是对这些位块传输进行特殊处理。如果在 CPU 上进行位块传输，对源值和目标值取平方根、执行线性混合函数、然后对结果取平方通常既快速又足够接近（近似目标编码的传递函数为平方）。许多 GPU 有一种可以在 sRGB 中进行混合的模式，但不幸的是这通常适用于整个帧缓冲区（framebuffer），而不仅仅是单次绘制。由于各种原因，Skia 避免了对这些混合进行特殊处理。

Skia 目前使用的是伽马技巧。创建字形蒙版时，通常知道将要绘制的大致颜色、传递函数，以及将使用线性 source-over 混合。然后猜测目标颜色为对比色（如果没有对比度，绘制将不可见），因此假设目标颜色将是目标色彩空间中感知上相反的颜色。现在可以通过将感知源色和猜测的目标色转换为线性编码、执行线性 source-over 混合并转换回目标编码来确定期望值。然后调整覆盖率，使得在非线性编码值上进行线性 source-over 混合的结果尽可能接近该期望值。

这种方法有效，但有很多假设。首先是对目标的猜测。感知上的中灰可以同样好（或差）地与黑色或白色形成对比，因此最好的猜测是将其绘制在自身之上。用这种猜测绘制的亚像素抗锯齿字形蒙版将完全不进行调整，使其产生色彩边缘。在 macOS 上，Skia 在使用明亮源色时会调整目标猜测以减少校正。这有助于减少亮色在暗色背景上（即暗色模式）的"光晕"问题，并更好地匹配 CoreText 的混合方式。第二个假设是假定使用 src-over 混合，但用户通常对其他混合方式的结果不那么挑剔。

伽马技巧在亚像素抗锯齿中效果最好，因为可以逐通道而非逐像素进行调整。如果将此技巧应用于全像素抗锯齿蒙版，则所有内容本质上都在最近的灰色而非最近的颜色中处理（因为只有一个通道可以用于传递信息），导致效果较差。大多数用户不希望将伽马技巧应用于全像素抗锯齿字形。

由于伽马技巧在逻辑上是混合的一部分，它必须始终是在位块传输器（blitter）使用覆盖率蒙版之前对其进行的最后一次调整。输入是目标传递函数和当前颜色（在目标色彩空间中）。在 Skia 中，这些来自 SkSurface 上的色彩空间和 SkPaint 上的颜色。


光学尺寸调整技巧（Optical Sizing Hack）
-------------------

在金属活字（metal type）时代，字体设计师会在纸上画出最终铅字字面的设计。不同的目标尺寸通常会有不同的绘图。有时这些不同尺寸看起来差异很大（如同一家族的展示体（Display）和正文体（Text）变体）。然后刻字工（punch cutter）会根据这些绘图创建一块称为字冲（punch）的金属片，用于冲压出字面。字冲用于在较软的金属片中创建一个阴模，形成字模（strike）。字模被制成正确的宽度，称为字模（matrix）。字模用作铸模来铸造单个铅字（sort）。铅字被收集到一个称为字盘（type case）的盒子中。排字工（typesetter）会从字盘中取出铅字并将它们排列成版面（form）。印刷工（printer）随后给版面中的铅字上墨并压印纸张。（注意这个描述中没有使用"typeface"和"font"这两个术语，关于它们适用的对象有很多争议。）

这一过程的每个步骤现在都以某种方式实现了自动化。不幸的是，嵌入在手工过程中的知识并未总是在自动化中得到复制。这可能有多种原因，例如被忽视或难以模拟。其中一个领域是光学尺寸调整和管理细笔画特征的艺术。

一般来说，较小的字号应该相对比线性缩小较大字号所预期的更重。字体设计师在绘图时会考虑到这一点，可能会考虑它如何随尺寸变化以及可能需要的墨水陷阱（ink trap）。刻字工随后会在给定尺寸上考虑这一点来刻制字冲，不断调整直到烟灰打样（soot proof）看起来良好。如果某些地方看起来不对，甚至可能对字模本身进行微调。排字工通常知道哪些特定的字盘包含略重或略轻的铅字。印刷工随后会调整墨量和压力以获得美观的印刷效果。

流行的数字字体格式直到最近才通过可变字体（variable font）的 'opsz' 轴真正支持这一功能。'opsz' 轴的存在就像字体设计师提供了关于字面在各种尺寸下应该如何呈现的非常好的指令。然而，并非所有字体都有或将会有 'opsz' 轴。既然我们并不总是有 'opsz'，那么如何防止小字形看起来模糊不清？

字体设计师影响光学尺寸的一种方式是通过微调（hinting）。任何手动微调都是由人在查看小尺寸下的结果时完成的。进行调整直到"看起来不错"。这通常会无意中融入光学尺寸调整。即使自动微调工具（autohinter）最终也会模拟这一点，通常在小像素尺寸下使文本更有对比度且更重（这取决于目标像素尺寸而非名义请求尺寸）。

设计师提供光学尺寸字体的另一种方式是为不同的光学尺寸使用多个字体文件，并期望用户选择正确的文件（如展示体和正文体变体）。理论上，也可以根据某些标准自动选择正确的文件。因为名义尺寸改变而切换字体文件可能看起来很极端，但这正是 macOS 上系统字体一度的工作方式。

一种自动进行光学尺寸调整的方法是在小名义尺寸下人为加粗轮廓本身。这是 CoreText 采用的方法。这很像令人诟病的假粗体（fake-bold），但没有自动假粗体比超粗体还重的问题，因为这种加粗是基于请求的名义尺寸统一应用的。

另一种自动进行光学尺寸调整的方法是在进行抗锯齿时过度覆盖所有边缘。像素是固定物理尺寸的，因此对小特征的影响大于大特征。Skia 目前通过其"对比度技巧"（contrast hack）对此提供了基本支持，稍后将进一步描述。

关于光学尺寸调整需要注意的一点是，这是字体的名义或光学尺寸与字体最终像素尺寸被区别对待的一个地方。SkCanvas 可能有一个活动的缩放变换。任何微调都将在最终像素尺寸（字体尺寸通过当前变换矩阵映射后）下完成，但光学尺寸不受变换影响，因为它实际上是 SkTypeface（也可能是 SkFont）的一个参数。


对比度技巧（The Contrast Hack）
-----------------

考虑 [半像素边界上一像素宽笔画](http://rastertragedy.com/RTRCh3.htm#Sec1) 的例子。如 3.1.3 节所述，"如果我们选择笔画位置以渲染最忠实的比例和步进宽度，我们可能不得不在渲染对比度上做出妥协。"如果笔画落在像素边界上，一切正常。如果笔画落在半像素边界上，且使用正确的线性混合，那么到达眼睛的光子总数相同，但光子的减少分布在两倍的面积上。如果像素足够小，这就不是问题。粗略估算表明 8K 20 英寸桌面显示器或约 400ppi 是最低要求。注意 RGB 亚像素抗锯齿将 100dpi 显示器提升到 300dpi，这很接近但仍然略有不足。当填充颜色接近 RGB 原色时，利用 RGB 亚像素实际上也不会那么多地增加分辨率。

一种尝试补偿的方法是增加这些部分覆盖像素的覆盖率，直到视觉效果更好。请注意，这取决于许多因素，如显示器的像素密度、用户的视觉敏锐度、用户与显示器的距离，以及用户对可能由此产生的笔画深浅变化的敏感度。自动考虑所有这些因素将相当困难。正确的函数通常是通过让用户查看应用不同额外覆盖量的结果，并让他们选择看起来最不差的设置来确定的。

这种特定形式的过度覆盖是抗锯齿的一种丢弃控制（drop out control）形式，可以以类似的方式实现：检测笔画何时在一个像素中出现并在下一个像素中消失，并标记为需要额外覆盖。在光栅化时，可以在水平和垂直方向上各做一遍，找到连续多个非完全覆盖像素的区域并增加其覆盖率，作为更粗略的近似。

如果不做这些计算开销昂贵的遍历而是提升所有覆盖率，那么除了模糊的笔画外，字形的整个外部边缘也会被加粗。使这些外部边缘更重是以一种相当复杂的方式外扩初始路径的粗略近似，相当于一种光学尺寸调整。就像微调可以在预先知道用户对像素尺寸的感知时用于近似光学尺寸调整一样，这是一种与特定用户和显示器组合相关的像素级调整。

就像伽马技巧可以被修改以减少亮色在暗色背景上的校正来对抗光晕效应一样，对比度技巧在已知绘制颜色为亮色时可以减弱。通常，对比度校正在接近白色时趋向于零。
