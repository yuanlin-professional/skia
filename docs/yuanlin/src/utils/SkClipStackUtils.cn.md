# SkClipStackUtils

> 源文件
> - src/utils/SkClipStackUtils.h
> - src/utils/SkClipStackUtils.cpp

## 概述

`SkClipStackUtils` 提供了用于处理裁剪栈（clip stack）的实用函数，主要功能是将复杂的裁剪栈解析为单一的路径表示。该工具模块简化了需要将多层裁剪操作合并为单个几何形状的场景，例如在 PDF 后端或其他矢量图形导出场景中。

## 架构位置

该模块位于 Skia 工具层（`src/utils/`），作为核心裁剪系统（`SkClipStack`）和路径操作库（`SkPathOps`）的桥接层。它主要服务于：
- 文档生成后端（如 PDF、XPS）
- 需要扁平化裁剪层次的图形导出场景
- 调试和可视化工具

## 主要类与结构体

本模块不定义类，仅提供独立的工具函数。主要使用的外部类型：

### 相关类型
- `SkClipStack`: 核心裁剪栈类，维护多层裁剪操作
- `SkClipStack::Element`: 裁剪栈的单个元素，包含几何形状和操作类型
- `SkClipStack::Iter`: 裁剪栈迭代器，支持从底向上或从顶向下遍历
- `SkPath`: 表示最终合并的裁剪路径
- `SkPathOps`: 路径布尔运算库

## 公共 API 函数

### SkClipStack_AsPath
```cpp
SkPath SkClipStack_AsPath(const SkClipStack& cs)
```

**功能：**
将整个裁剪栈解析为单一路径，通过路径布尔运算逐层合并所有裁剪元素。

**参数：**
- `cs`: 输入的裁剪栈引用

**返回值：**
合并后的 `SkPath` 对象，填充规则设置为 `kInverseEvenOdd`

**处理逻辑：**
1. 创建初始路径，设置反向偶奇填充规则
2. 从底向上遍历裁剪栈
3. 对每个元素：
   - 跳过着色器类型的裁剪（无法转换为路径）
   - 获取元素的设备空间路径表示
   - 根据操作类型（交集、并集等）应用路径布尔运算
   - 特殊处理替换操作（直接覆盖当前路径）

## 内部实现细节

### 遍历策略
使用 `kBottom_IterStart` 从栈底开始遍历，按照裁剪操作的历史顺序处理。这确保了操作的正确累积顺序。

### 裁剪元素类型处理

**1. 空元素（Empty）**
空裁剪区域，创建空的 `operand` 路径。

**2. 着色器元素（Shader）**
着色器裁剪无法表示为几何路径，当前实现跳过这些元素。代码中标记为 TODO，暗示未来可能在特定后端（如 PDF）中以其他方式处理。

**3. 几何元素**
调用 `asDeviceSpacePath()` 获取路径表示，支持矩形、圆角矩形、路径等几何类型。

### 操作类型处理

**1. 替换操作（Replace）**
- 直接用新路径替换整个裁剪区域
- 代码中注释表明，一旦扩展裁剪操作被移除，可以改为从顶向下迭代并在遇到替换操作时提前退出

**2. 布尔操作**
- 使用 `SkPathOps::Op()` 执行路径布尔运算
- 支持的操作类型由 `SkClipOp` 定义（交集、差集、并集等）
- 操作结果可能失败（返回空 optional），此时保持原路径不变

### 填充规则选择
初始路径使用 `kInverseEvenOdd` 填充规则，这是一种常见的裁剪合并策略：
- 反向（Inverse）：路径外部被视为填充区域
- 偶奇（EvenOdd）：交叉次数的奇偶性决定内外

## 设计模式与设计决策

### 函数式 API 设计
提供纯函数接口而非类封装，符合工具函数的简洁性要求。函数接受 const 引用，不修改输入，返回新对象。

### 渐进式累积策略
通过迭代和布尔运算逐步构建最终路径，而非一次性计算。这种方法虽然复杂度较高，但逻辑清晰且易于调试。

### 容错设计
对无法处理的元素（如着色器裁剪）采用跳过策略，保证函数在各种输入下的健壮性。

### 未来扩展性
代码中包含多处 TODO 注释，表明设计考虑了未来的优化方向：
- 支持着色器裁剪的特殊处理
- 优化迭代方向以利用替换操作
- 移除已废弃的扩展裁剪操作

## 性能考量

### 时间复杂度
- O(n * k)，其中 n 是裁剪栈元素数量，k 是路径布尔运算的平均复杂度
- 路径布尔运算是计算密集型操作，涉及边缘求交和拓扑排序

### 空间复杂度
- O(m)，其中 m 是最终路径的复杂度（顶点和边的数量）
- 每次布尔运算可能产生临时路径对象

### 性能限制
1. **不适合实时渲染**：路径布尔运算开销大，不应在每帧调用
2. **适合离线导出**：主要用于文档生成等一次性转换场景
3. **复杂裁剪栈代价高**：元素越多，累积的布尔运算次数越多

### 优化建议
- 对于简单裁剪栈（单个矩形等），可以考虑快速路径
- 缓存结果以避免重复计算
- 在可能的情况下，提前简化裁剪栈
