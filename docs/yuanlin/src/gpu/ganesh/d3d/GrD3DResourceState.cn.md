# GrD3DResourceState

> 源文件
> - `src/gpu/ganesh/d3d/GrD3DResourceState.h`

## 概述

`GrD3DResourceState` 是一个轻量级的线程安全类,用于跟踪 Direct3D 12 资源的状态。D3D12 要求应用程序显式管理资源状态转换(通过资源屏障),该类封装了当前资源状态的存储和同步访问,支持多个对象安全地共享同一资源的状态信息。

该类使用原子变量存储 `D3D12_RESOURCE_STATES` 枚举值,提供线程安全的读写操作。通过引用计数(`SkRefCnt`)支持多个纹理、渲染目标或其他资源对象共享同一状态跟踪器,确保状态更新对所有引用者可见。

## 架构位置

```
Skia GPU Backend (Ganesh)
└── Direct3D 12 资源管理
    ├── GrD3DTextureResource (使用状态跟踪)
    ├── GrD3DTexture (共享状态)
    ├── GrD3DRenderTarget (共享状态)
    └── GrD3DResourceState (当前类 - 状态跟踪)
```

该类是 D3D12 资源管理的基础设施组件,为所有 D3D12 资源提供状态同步机制。

## 主要类与结构体

### GrD3DResourceState

**继承关系**
- 继承自: `SkRefCnt` - 提供引用计数和智能指针支持

**关键成员变量**

| 成员变量 | 类型 | 说明 |
|---------|------|------|
| `fState` | `std::atomic<D3D12_RESOURCE_STATES>` | 原子变量存储的资源状态标志位 |

**设计特点:**
- 极简设计,仅 32 行代码
- 仅包含单一原子成员变量
- 所有方法内联实现

## 公共 API 函数

### 构造函数

```cpp
GrD3DResourceState(D3D12_RESOURCE_STATES state);
```

使用指定的初始状态创建状态跟踪器。

**参数:** `state` - D3D12 资源状态标志位,如 `D3D12_RESOURCE_STATE_COMMON`、`D3D12_RESOURCE_STATE_RENDER_TARGET` 等。

### 状态设置

```cpp
void setResourceState(D3D12_RESOURCE_STATES state);
```

更新资源状态。使用顺序一致性内存顺序(`std::memory_order_seq_cst`)保证跨线程可见性。

**线程安全:** 可以从多个线程安全调用。

### 状态获取

```cpp
D3D12_RESOURCE_STATES getResourceState() const;
```

读取当前资源状态。同样使用顺序一致性内存顺序保证最新值的可见性。

**线程安全:** 可以从多个线程安全调用。

## 内部实现细节

### 原子操作语义

使用 `std::atomic<D3D12_RESOURCE_STATES>`:
- **类型:** `D3D12_RESOURCE_STATES` 是 32 位整数类型,原子操作高效
- **内存顺序:** 默认使用 `memory_order_seq_cst`(顺序一致性)
- **保证:** 所有线程看到一致的状态更新顺序

### 顺序一致性选择

代码注释明确说明使用顺序一致性:
```cpp
// Defaulting to use std::memory_order_seq_cst
```

**权衡:**
- **优点:** 最强的内存顺序保证,简化推理
- **缺点:** 性能略低于弱内存顺序
- **结论:** 对于状态管理的关键性,选择正确性优先

### 引用计数共享

通过继承 `SkRefCnt`:
- 支持 `sk_sp<GrD3DResourceState>` 智能指针
- 自动管理生命周期
- 多个资源对象可以指向同一状态跟踪器

### 状态共享的使用场景

多个对象共享状态跟踪器的情况:
1. **别名资源**: 多个视图指向同一 D3D12 资源
2. **包装纹理**: 原始纹理和包装对象
3. **纹理渲染目标**: 纹理视图和渲染目标视图

### 最小化设计哲学

该类刻意保持极简:
- 仅存储单一状态值
- 没有复杂的状态转换逻辑
- 状态管理策略由使用者决定

这种设计使得类易于理解、测试和正确使用。

### 内联实现

所有方法都在头文件中定义:
- 支持编译器内联优化
- 减少函数调用开销
- 对于如此简单的方法,内联是理想选择

## 设计模式与设计决策

### 单一职责原则

类仅负责存储和原子访问资源状态:
- 不包含状态转换逻辑
- 不生成资源屏障
- 不验证状态有效性

这使得类极其简单和可靠。

### 引用计数共享

使用引用计数而非垃圾回收:
- 确定性的生命周期管理
- 低开销(相比 GC)
- 与 Skia 的智能指针体系一致

### 强一致性内存模型

选择最强的内存顺序:
- 避免微妙的并发错误
- 简化正确性推理
- 性能损失在可接受范围内

对于资源状态这种关键数据,正确性优先于极致性能。

### 不可变 vs 可变

状态对象本身是可变的(允许状态更新):
- 支持资源状态的动态转换
- 反映 D3D12 资源状态的可变性
- 通过原子操作保证并发安全

### 类型安全封装

封装原子变量而非直接暴露:
- 控制访问方式(通过方法)
- 保证使用正确的内存顺序
- 未来可以修改内部实现而不破坏 API

## 性能考量

### 原子操作开销

顺序一致性原子操作的成本:
- **x86/x64**: 相对便宜,主要是编译器屏障
- **ARM**: 可能需要内存屏障指令
- **频率**: 状态转换不频繁,开销可接受

### 缓存一致性

原子变量的缓存行为:
- 读操作可能从缓存读取
- 写操作需要缓存一致性协议
- 对于共享频繁的状态,可能导致缓存抖动

### 引用计数开销

`sk_sp` 的引用计数操作:
- 每次拷贝/赋值: 原子递增/递减
- 相比深拷贝状态跟踪器,这是巨大的节省
- 移动语义可以避免引用计数操作

### 内联优化

所有方法内联后:
- `getResourceState`: 编译为单个原子加载指令
- `setResourceState`: 编译为单个原子存储指令
- 无函数调用开销

### 状态读取模式

频繁读取、偶尔写入的访问模式:
- 原子加载在 x86 上基本无成本(仅编译器屏障)
- 写入时的同步成本被摊销
- 符合状态跟踪的典型使用模式

### 对象大小

极小的对象大小:
- 仅包含一个 `std::atomic<uint32_t>` (4 字节)
- 加上虚表指针(8 字节,来自 `SkRefCnt`)
- 总计 16 字节(考虑对齐),非常缓存友好
