# find_headers.py

> 源文件: gn/find_headers.py

## 概述

`find_headers.py` 是 Skia 构建系统中的一个关键 Python 脚本,用于自动生成统一的头文件 `skia.h`。该脚本通过 GN 构建系统查询所有公共头文件,并将它们汇总到单个头文件中,同时生成 Ninja 构建系统所需的依赖关系文件。这个脚本确保 Skia 的公共 API 可以通过单一入口点访问,简化了外部项目的集成工作。

该脚本的主要功能包括:从 GN 构建描述中提取公共头文件列表,去重和规范化头文件路径,生成格式化的 `skia.h` 文件,以及创建 Ninja 依赖跟踪文件 `skia.h.deps`。

## 架构位置

`find_headers.py` 位于 Skia 项目的构建工具层:

```
skia/
├── gn/                          # GN 构建脚本目录
│   ├── find_headers.py          # 本脚本 - 头文件收集器
│   ├── gn_to_bp.py              # Android 构建文件生成器
│   └── ...                      # 其他构建辅助脚本
├── include/                     # 公共头文件目录
│   ├── core/                    # 核心 API 头文件
│   ├── gpu/                     # GPU API 头文件
│   └── ...
└── out/                         # 构建输出目录
    └── <config>/
        └── skia.h               # 生成的统一头文件
```

在构建流程中的位置:
- **输入**: GN 构建配置、源代码目录结构
- **处理**: 查询 GN 元数据、过滤和排序头文件
- **输出**: `skia.h` 统一头文件、`skia.h.deps` 依赖文件

## 主要类与结构体

### Header (namedtuple)
```python
Header = collections.namedtuple('Header', ['absolute', 'include'])
```

**用途**: 存储头文件的双重路径表示

**字段**:
- `absolute`: 头文件的绝对文件系统路径
- `include`: 相对于 include 目录的包含路径(用于 `#include` 语句)

**设计意图**: 使用 `namedtuple` 提供轻量级的不可变数据结构,既有元组的性能,又有字段名的可读性。

## 公共 API 函数

脚本作为命令行工具运行,通过命令行参数接收配置:

### 命令行接口
```python
# 参数顺序:

# sys.argv[2] - 源代码根目录的绝对路径
# sys.argv[3] - 输出的 skia.h 文件路径
# sys.argv[4:] - include 目录列表
```

**调用示例**:
```bash
python gn/find_headers.py gn /path/to/skia out/skia.h include/ include/core/ include/gpu/
```

### 核心处理流程

1. **GN 查询阶段**:
```python
gn_desc_cmd = [gn, 'desc', '.', '--root=%s' % absolute_source, '--format=json', '*']
desc_json_txt = subprocess.check_output(gn_desc_cmd).decode('utf-8')
```
执行 `gn desc` 命令获取所有目标的 JSON 格式构建描述。

2. **头文件提取**:
```python
for target in desc_json.values():
    search_list = target.get('public')
    if search_list == '*':
        search_list = target.get('sources', [])
    for name in search_list:
        sources.add(os.path.join(absolute_source, os.path.normpath(name[2:])))
```
从每个构建目标中提取 `public` 字段声明的头文件,如果没有显式声明,则从 `sources` 中提取。

3. **路径规范化**:
```python
for source in sources:
    source_as_include = [os.path.relpath(source, absolute_source)
                         for include_dir in include_dirs
                         if source.startswith(include_dir)]
```
将绝对路径转换为相对于 include 目录的路径。

4. **文件去重**:
```python
statinfo = os.stat(source)
key = str(statinfo.st_ino) + ':' + str(statinfo.st_dev)
if key == '0:0':  # Windows 兼容性处理
    key = source
if key not in headers or len(include_path) < len(headers[key].include):
    headers[key] = Header(source, include_path)
```
使用 inode 和设备号作为唯一标识符去重,Windows 系统回退到路径比较。

## 内部实现细节

### 错误处理机制

**GN 警告检测**:
```python
if desc_json_txt.startswith('WARNING'):
    print('\ngn generated a warning when we asked for JSON output.', ...)
    sys.exit(-1)
```
GN 可能在 JSON 输出前插入警告信息,脚本检测此情况并提示用户查看完整警告。

**JSON 解析保护**:
```python
try:
    desc_json = json.loads(desc_json_txt)
except ValueError:
    print(desc_json_txt)
    raise
```
捕获 JSON 解析错误并输出原始内容以帮助调试。

### 路径处理策略

**Include 目录排序**:
```python
include_dirs.sort(key=len, reverse=True)
```
按长度降序排序 include 目录,确保更具体的路径(如 `include/core/`)优先匹配,避免被通用路径(如 `include/`)覆盖。

**路径截断处理**:
```python
name[2:]  # 去掉 GN 路径前缀 "//"
```
GN 内部路径以 `//` 开头表示源码根目录,需要截断后才能拼接成有效的文件系统路径。

**最短路径选择**:
```python
if key not in headers or len(include_path) < len(headers[key].include):
    headers[key] = Header(source, include_path)
```
当同一文件通过多个 include 路径可达时,选择最短路径,提高可读性和编译效率。

### 输出文件生成

**skia.h 生成**:
```python
with open(skia.h, 'w') as f:
    f.write('// skia.h generated by GN.\n')
    f.write('#ifndef skia_h_DEFINED\n')
    f.write('#define skia_h_DEFINED\n')
    for header in headers:
        f.write('#include "' + header.include + '"\n')
    f.write('#endif//skia_h_DEFINED\n')
```
生成标准的 C++ 头文件保护宏,并按排序顺序包含所有公共头文件。

**依赖文件生成**:
```python
with open(skia_h + '.deps', 'w') as f:
    f.write(skia_h + ':')
    for header in headers:
        f.write(' ' + header.absolute)
    f.write(' build.ninja.d')
    f.write('\n')
```
生成 Makefile 风格的依赖规则,格式为 `target: dependency1 dependency2 ...`,Ninja 使用此文件跟踪头文件变更。

### Windows 兼容性

```python
# On Windows os.stat st_ino is 0 until 3.3.4 and st_dev is 0 until 3.4.0.
if key == '0:0':
    key = source
```
旧版 Python 在 Windows 上不提供有效的 inode 信息,脚本回退到使用完整路径作为唯一标识符。

### 临时文件清理

```python
if os.path.exists(skia_h + '.d'):
    os.remove(skia_h + '.d')
```
清理旧版本脚本生成的 `.d` 文件,确保构建环境的一致性。

## 设计模式与设计决策

### 单一职责原则
脚本专注于头文件聚合任务,不处理编译、链接或其他构建步骤,保持工具的简洁性和可维护性。

### 数据驱动设计
依赖 GN 提供的结构化元数据,而非硬编码头文件列表,确保与构建系统的自动同步。

### 防御性编程

**多层错误处理**:
- subprocess 异常捕获
- JSON 格式验证
- 文件系统访问错误处理

**平台兼容性检查**:
```python
if key == '0:0':  # Windows 特殊处理
```
主动检测平台差异并提供回退方案。

### 声明式输出格式
生成的 `skia.h` 使用声明式的 `#include` 列表,而非复杂的逻辑,便于人类阅读和工具处理。

通过 `.deps` 文件为 Ninja 提供细粒度依赖信息,仅在头文件变更时重新生成 `skia.h`,优化构建性能。

## 性能考量

### 文件系统访问优化

**批量查询**:
```python
desc_json = json.loads(desc_json_txt)  # 单次解析
```
通过 `gn desc *` 一次性查询所有目标,避免多次调用 GN。

**延迟文件访问**:
```python
os.stat(source)  # 仅在需要去重时访问文件系统
```
只有在生成最终列表时才进行 stat 调用,减少 I/O 操作。

### 数据结构选择

**set 用于去重**:
```python
sources = set()  # O(1) 插入和查找
```
使用集合存储源文件路径,提供高效的去重操作。

**字典用于唯一性保证**:
```python
headers = {}  # key-value 映射,O(1) 查找
```
使用字典按 inode 索引头文件,快速检测重复。

### 排序策略

**单次排序**:
```python
headers = sorted(headers.values(), key=lambda x: x.include)
```
在所有处理完成后执行一次排序,避免重复排序开销。

**排序键选择**:
使用 include 路径排序,确保生成的头文件顺序稳定且有意义。

### 内存管理

**生成器避免使用**: 虽然 Python 生成器可以节省内存,但脚本需要完整的文件列表进行去重和排序,因此使用集合和列表。

**及时释放**: subprocess 输出在解析后不再保留,减少内存占用。
