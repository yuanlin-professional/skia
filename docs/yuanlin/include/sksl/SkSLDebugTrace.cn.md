# SkSL 调试跟踪

> 源文件: `include/sksl/SkSLDebugTrace.h`

## 概述

`DebugTrace` 是 SkSL (Skia Shading Language) 编译器的调试追踪接口,用于在着色器编译和执行过程中收集调试信息。该类提供了一个抽象接口,允许调试工具生成人类可读的调试跟踪转储,帮助开发者诊断着色器问题。

## 架构位置

本模块位于 Skia 的 SkSL 子系统中,属于着色器语言编译基础设施层。它为 SkSL 编译器提供了调试和诊断能力,是着色器开发和调试工具链的重要组成部分。

## 主要类与结构体

### DebugTrace

**职责**: 作为调试跟踪的抽象基类,定义了调试信息转储的统一接口。

**继承关系**: `SkRefCnt` → `DebugTrace`

该类继承自 `SkRefCnt`,使用引用计数管理对象生命周期,确保调试跟踪对象可以在多个组件之间安全共享。

**关键设计特点**:
- 纯虚类设计,强制子类实现具体的调试跟踪机制
- 基于引用计数的生命周期管理
- 提供统一的调试信息输出接口

## 公共 API 函数

### `virtual void dump(SkWStream* o) const = 0`

- **功能**: 生成人类可读的调试跟踪转储并写入输出流
- **参数**:
  - `o`: 指向 `SkWStream` 的指针,用于接收调试信息输出
- **返回值**: 无返回值
- **特性**: 纯虚函数,必须由子类实现具体的转储逻辑

该函数是 `DebugTrace` 接口的核心方法,它允许调试工具获取格式化的调试信息。输出流的设计使得调试信息可以灵活地输出到文件、内存缓冲区或其他目标。

## 内部实现细节

### 抽象接口设计

`DebugTrace` 采用抽象基类模式,不包含具体实现。这种设计允许不同的 SkSL 编译后端实现各自的调试跟踪机制:

1. **可扩展性**: 不同的着色器目标(GLSL、SPIR-V、Metal 等)可以实现特定于平台的调试信息
2. **解耦设计**: 调试跟踪接口与具体编译器实现分离,提高代码模块化
3. **统一访问**: 无论底层实现如何,所有调试跟踪都通过相同的 `dump` 接口访问

### 生命周期管理

通过继承 `SkRefCnt`,调试跟踪对象使用引用计数自动管理内存:

```cpp
sk_sp<DebugTrace> trace = CreateDebugTrace();  // 引用计数为 1
trace->dump(&stream);                           // 使用跟踪对象
// 离开作用域时自动释放
```

## 设计模式与设计决策

### 策略模式 (Strategy Pattern)

`DebugTrace` 使用策略模式定义调试跟踪算法的接口。不同的具体实现类可以提供不同的调试信息格式和详细程度,而客户端代码统一通过 `dump()` 方法访问。

### 接口隔离原则

该接口非常精简,只包含一个纯虚方法。这遵循了接口隔离原则,确保实现类只需关注调试信息的生成和格式化,而不承担其他职责。

### 依赖倒置原则

高层模块(如调试工具)依赖于 `DebugTrace` 抽象接口,而不是具体实现。这使得可以在不修改客户端代码的情况下替换不同的调试跟踪实现。

## 使用场景

### 着色器开发调试

开发者在编写 SkSL 着色器时,可以启用调试跟踪来:
- 查看编译器生成的中间表示
- 跟踪变量值的变化
- 识别编译警告和错误的来源

### 性能优化

性能分析工具可以使用调试跟踪来:
- 识别着色器中的性能热点
- 分析指令执行次数
- 评估不同优化策略的效果

### 单元测试

SkSL 编译器的单元测试可以使用调试跟踪来:
- 验证编译器生成正确的代码
- 比较不同版本的编译器输出
- 回归测试调试信息格式

## 扩展点

虽然当前接口非常简单,但未来可以扩展以支持:

1. **结构化查询**: 添加方法以编程方式查询特定调试信息
2. **过滤器**: 支持选择性转储特定类型的调试信息
3. **格式选项**: 支持多种输出格式(JSON、XML、纯文本等)
4. **实时跟踪**: 支持流式调试信息输出,而不仅仅是批量转储

## 总结

`DebugTrace` 是一个设计精良的抽象接口,它为 SkSL 着色器语言的调试和诊断提供了统一的入口点。通过简洁的接口和灵活的实现机制,它支持多种调试场景,从交互式开发到自动化测试。其基于引用计数的生命周期管理和抽象接口设计使其能够无缝集成到 Skia 的编译基础设施中。
