# SkPixelStorage

> 源文件: `include/private/SkPixelStorage.h`

## 概述

SkPixelStorage 是 Skia 图形库中表示像素数据存储块的抽象基类。它作为像素内存的抽象来源,为 SkImage 和 SkSurface 等高层对象提供可读写的二维像素数据数组。该类是 2026 年新增的重要架构改进,用于统一管理不同类型的像素存储。

## 架构位置

SkPixelStorage 位于 Skia 核心架构的底层存储层,作为像素数据的统一抽象接口。它是 SkPixelRef 和 TextureProxy 的共同基类,在图像和纹理系统之间提供了一致的接口抽象。该类定义在 `include/private` 目录中,属于 Skia 内部 API。

## 主要类与结构体

### SkPixelStorage

SkPixelStorage 是一个抽象基类,代表所有像素存储实现的统一接口。它为不同类型的像素存储提供了类型识别和唯一标识符管理功能。

**继承关系**: `SkPixelStorage` (抽象基类)

**关键成员变量**:

| 变量名 | 类型 | 说明 |
|--------|------|------|
| fID | uint32_t | 存储对象的唯一标识符,用于追踪和缓存 |

### Type 枚举

定义了像素存储的两种类型:

| 枚举值 | 说明 |
|--------|------|
| kTextureProxy | GPU 纹理代理类型,用于 GPU 加速渲染 |
| kPixelRef | CPU 像素引用类型,用于 CPU 端像素数据 |

## 公共 API 函数

### `SkPixelStorage()`
- **功能**: 默认构造函数,创建一个新的像素存储对象并分配唯一 ID
- **参数**: 无
- **返回值**: 无

### `virtual ~SkPixelStorage()`
- **功能**: 虚析构函数,确保派生类能够正确清理资源
- **参数**: 无
- **返回值**: 无
- **说明**: 使用 default 实现,派生类可以覆盖以添加自定义清理逻辑

### `uint32_t getId() const`
- **功能**: 获取当前存储对象的唯一标识符
- **参数**: 无
- **返回值**: uint32_t 类型的唯一 ID
- **说明**: 该 ID 在对象生命周期内保持不变,可用于缓存键和追踪

### `virtual Type type() const`
- **功能**: 获取像素存储的具体类型(纯虚函数)
- **参数**: 无
- **返回值**: Type 枚举值(kTextureProxy 或 kPixelRef)
- **说明**: 派生类必须实现此方法来标识自己的类型

## 内部实现细节

### ID 生成机制

SkPixelStorage 使用静态函数 `NextId()` 来生成全局唯一的 ID。这个机制确保每个像素存储对象都有一个独一无二的标识符,用于以下场景:

1. **缓存管理**: ID 可作为缓存系统的键,实现高效的查找和失效处理
2. **对象追踪**: 在调试和性能分析中追踪特定的像素存储对象
3. **引用比较**: 快速判断两个指针是否指向同一个存储对象

### 类型系统设计

通过 Type 枚举和纯虚函数 `type()`,SkPixelStorage 实现了运行时类型识别(RTTI)。这种设计允许:

- **类型安全的向下转型**: 在需要访问派生类特定功能时,可以先检查类型再进行转换
- **分支优化**: 渲染管线可以根据存储类型选择不同的处理路径
- **统一接口**: 上层代码可以通过基类指针统一处理不同的存储类型

## 依赖关系

### 依赖的模块

| 依赖 | 用途 |
|------|------|
| `<cstdint>` | 提供 uint32_t 等标准整数类型定义 |

### 被依赖的模块

SkPixelStorage 作为基类,被以下模块依赖:

- **SkPixelRef**: CPU 端像素数据的引用计数管理类,继承自 SkPixelStorage
- **TextureProxy**: GPU 端纹理代理类,继承自 SkPixelStorage,用于延迟纹理创建
- **SkImage**: 图像类使用 SkPixelStorage 作为底层像素数据的抽象接口
- **SkSurface**: 绘图表面类通过 SkPixelStorage 访问底层像素缓冲区

## 设计模式与设计决策

### 抽象工厂模式

SkPixelStorage 作为抽象基类,体现了抽象工厂模式的思想。不同的派生类(SkPixelRef 和 TextureProxy)作为具体工厂,提供不同的像素存储实现。

### 模板方法模式

通过纯虚函数 `type()`,定义了类型查询的接口规范,由派生类提供具体实现。这是模板方法模式的一种应用。

### 设计决策说明

1. **为什么使用 ID 而不是指针比较**:
   - ID 在对象销毁后仍然有效,可用于检测悬空引用
   - 在多线程环境中,ID 比较比指针比较更安全
   - ID 可以序列化,便于调试和日志记录

2. **为什么使用枚举而不是 dynamic_cast**:
   - 避免 RTTI 的性能开销
   - 在禁用 RTTI 的编译配置下仍然可用
   - 提供更快的类型检查路径

3. **为什么设计为抽象基类**:
   - 统一 CPU 和 GPU 像素存储的接口
   - 简化上层代码,无需区分不同的存储类型
   - 为未来扩展新的存储类型提供灵活性

## 性能考量

### ID 生成性能

静态函数 `NextId()` 的实现需要考虑多线程安全性。通常使用原子操作来保证 ID 的唯一性,这会带来一定的性能开销,但由于像素存储对象的创建频率相对较低,这个开销是可以接受的。

### 虚函数调用开销

`type()` 是虚函数,会带来虚函数调用的开销。但考虑到:
- 类型查询通常只在需要向下转型时进行
- 相比实际的像素数据处理,虚函数调用的开销可以忽略不计
- 编译器在某些情况下可以进行去虚拟化优化

### 内存占用

SkPixelStorage 对象的内存占用非常小:
- fID: 4 字节
- 虚函数表指针: 8 字节(64 位系统)
- 总计约 12-16 字节(考虑对齐)

这个开销相对于实际的像素数据来说微不足道。
