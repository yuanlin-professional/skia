# SkSafe32 - 32位安全整数运算

> 源文件: `include/private/base/SkSafe32.h`

## 概述

SkSafe32 提供了一组用于32位整数安全算术运算的函数，主要处理溢出饱和计算和避免未定义行为。该模块是 Skia 基础设施层的一部分，用于确保整数运算的安全性和可预测性。

## 架构位置

- **所属子系统**: 基础工具库 (Base Utilities)
- **层级**: 私有头文件，位于 `include/private/base/` 目录
- **依赖层次**: 底层基础模块，被上层数值计算模块依赖

## 核心功能函数

### `Sk64_pin_to_s32`

```cpp
static constexpr int32_t Sk64_pin_to_s32(int64_t x)
```

- **功能**: 将64位整数钳制到32位有符号整数的范围内
- **参数**:
  - `x`: 需要钳制的64位整数值
- **返回值**: 钳制后的32位整数，范围为 [SK_MinS32, SK_MaxS32]
- **实现原理**:
  - 如果 x < SK_MinS32，返回 SK_MinS32
  - 如果 x > SK_MaxS32，返回 SK_MaxS32
  - 否则返回 (int32_t)x

### `Sk32_sat_add`

```cpp
static constexpr int32_t Sk32_sat_add(int32_t a, int32_t b)
```

- **功能**: 饱和加法运算，防止32位整数加法溢出
- **参数**:
  - `a`: 第一个加数
  - `b`: 第二个加数
- **返回值**: 饱和计算结果，若溢出则返回 SK_MaxS32 或 SK_MinS32
- **实现原理**: 先转换为64位进行加法，然后通过 `Sk64_pin_to_s32` 钳制到32位范围

### `Sk32_sat_sub`

```cpp
static constexpr int32_t Sk32_sat_sub(int32_t a, int32_t b)
```

- **功能**: 饱和减法运算，防止32位整数减法溢出
- **参数**:
  - `a`: 被减数
  - `b`: 减数
- **返回值**: 饱和计算结果，若溢出则返回 SK_MaxS32 或 SK_MinS32
- **实现原理**: 先转换为64位进行减法，然后通过 `Sk64_pin_to_s32` 钳制到32位范围

### `Sk32_can_overflow_add`

```cpp
static constexpr int32_t Sk32_can_overflow_add(int32_t a, int32_t b)
```

- **功能**: 允许溢出的加法运算，通过无符号转换避免 UBSAN 警告
- **参数**:
  - `a`: 第一个加数
  - `b`: 第二个加数
- **返回值**: 二进制补码加法结果（可能溢出）
- **设计目的**: 用于故意进行溢出计算的场景，通过无符号运算避免未定义行为检测工具的警告

### `Sk32_can_overflow_sub`

```cpp
static constexpr int32_t Sk32_can_overflow_sub(int32_t a, int32_t b)
```

- **功能**: 允许溢出的减法运算，通过无符号转换避免 UBSAN 警告
- **参数**:
  - `a`: 被减数
  - `b`: 减数
- **返回值**: 二进制补码减法结果（可能溢出）
- **设计目的**: 用于故意进行溢出计算的场景

### `SkAbs32`

```cpp
static inline int32_t SkAbs32(int32_t value)
```

- **功能**: 32位整数的安全绝对值计算
- **参数**:
  - `value`: 需要计算绝对值的32位整数
- **返回值**: value 的绝对值
- **安全性**:
  - 通过断言检查 `value != SK_NaN32`
  - SK_NaN32 是最小的32位整数 (INT_MIN)，无法取反
  - 这与通用的 SkTAbs 函数不同，后者不进行此断言检查

## 内部实现细节

### 饱和运算的实现策略

饱和运算通过以下步骤实现：
1. 将32位操作数提升为64位
2. 在64位精度下执行运算，避免32位溢出
3. 使用 `Sk64_pin_to_s32` 将结果钳制回32位范围

### UBSAN 兼容性处理

`Sk32_can_overflow_*` 系列函数的设计考虑：
- C++ 标准中有符号整数溢出是未定义行为
- UBSAN (Undefined Behavior Sanitizer) 会检测并报告这类问题
- 通过转换为无符号类型进行运算，利用无符号整数的二进制补码溢出定义良好的特性
- 再转换回有符号类型，结果是实现定义的但可预测的

## 设计模式与设计决策

### 使用 constexpr 函数

所有函数都声明为 `constexpr`，允许编译期求值：
- 提高性能，常量表达式可在编译期计算
- 支持模板元编程
- 保持代码简洁，同时支持运行时和编译期使用

### 饱和算术的设计动机

饱和算术在图形处理中特别有用：
- 颜色值计算需要保持在 [0, 255] 范围
- 坐标计算溢出时钳制到边界比回绕更合理
- 避免溢出导致的视觉瑕疵

### 显式溢出支持

提供 `can_overflow` 系列函数的原因：
- 某些算法故意使用溢出特性（如哈希计算）
- 避免工具误报，明确标识这是有意为之
- 保持代码可移植性和清晰度

## 性能考量

### 内联和 constexpr

- 所有函数都是内联的，避免函数调用开销
- constexpr 允许编译器进行更积极的优化
- 常量参数调用可完全在编译期计算

### 饱和运算的成本

- 饱和加减法需要64位运算和钳制操作
- 成本略高于普通32位运算
- 权衡：安全性 vs 性能，适用于关键但非高频路径

### 与标准库的比较

- 比显式检查后条件跳转更高效
- 现代编译器可生成条件移动指令优化钳制操作
- 比异常处理方式高效得多

## 使用示例场景

### 坐标计算

```cpp
int32_t newX = Sk32_sat_add(oldX, deltaX);  // 防止坐标溢出
```

### 颜色混合

```cpp
int32_t blendedColor = Sk32_sat_add(color1, color2);  // 颜色值不会超出范围
```

### 尺寸计算

```cpp
int32_t totalSize = Sk32_sat_add(width, padding);  // 避免尺寸溢出
```

## 注意事项

1. **SkAbs32 的限制**: 不能对 SK_NaN32 (INT_MIN) 调用，否则会触发断言失败
2. **饱和行为**: 饱和运算返回边界值而非回绕，这可能不适合所有算法
3. **性能敏感**: 在极高频调用路径中，应评估饱和运算的性能影响
4. **UBSAN 环境**: 在启用 UBSAN 的构建中，优先使用这些安全函数而非原生运算符
