# mockery 资产创建脚本

> 源文件: infra/bots/assets/mockery/create.py

## 概述

这是一个用于从源码构建并创建 mockery 工具资产的 Python 脚本。mockery 是一个 Go 语言的 mock 生成工具，用于为接口生成测试模拟对象。该脚本专门构建 mockery v2.4.0 版本，并针对 Go 1.18 的兼容性问题进行了特殊修复。

## 架构位置

该脚本位于 Skia 项目的基础设施资产管理系统中，具体路径为 `infra/bots/assets/mockery/`。它是 Skia CI/CD 系统的一部分，负责准备构建和测试所需的第三方工具。mockery 资产被用于生成 Go 语言代码的测试模拟对象，支持 Skia 的 Go 语言基础设施代码的单元测试。

## 主要类与结构体

该脚本采用简单的函数式编程风格，没有定义类或复杂的数据结构。主要使用标准库模块：

- **argparse**: 用于命令行参数解析
- **subprocess**: 用于执行外部命令（git、go 等）
- **os**: 用于文件系统操作和路径管理
- **shutil**: 用于高级文件操作（复制、删除目录树）

脚本中定义了几个模块级常量：
- `REPO_URL`: mockery 的 GitHub 仓库地址
- `REPO_DIR`: 本地克隆仓库的目录名
- `BINARY_NAME`: 构建产物的二进制文件名

## 公共 API 函数

### `create_asset(target_dir)`

主要的资产创建函数，执行完整的构建流程。

**参数**:
- `target_dir` (str): 存放最终构建产物的目标目录

**功能流程**:
1. 克隆 mockery 的 GitHub 仓库到本地
2. 检出指定版本标签 v2.4.0
3. 更新 `golang.org/x/tools` 依赖以修复 Go 1.18 兼容性问题
4. 执行 `go mod tidy` 清理依赖
5. 使用特定的链接器标志构建 mockery 二进制文件
6. 将构建产物复制到目标目录
7. 清理临时文件和克隆的仓库

### `main()`

脚本入口函数，负责参数解析和函数调用。

**参数解析**:
- `--target_dir, -t`: 必需参数，指定资产输出目录

## 内部实现细节

### Go 1.18 兼容性修复

脚本中包含了针对 Go 1.18 的特殊修复逻辑。mockery v2.4.0 到 v2.10.0 在 Go 1.18 下会出现类型信息缺失的错误：
```
internal error: package "fmt" without types was imported from ...
```

解决方案是将 `golang.org/x/tools` 更新到 v0.1.10 版本。相关问题讨论见 GitHub issues #434 和 golang/go#49608。

### 链接器标志

构建时使用特定的链接器标志：
```bash
-s -w -X github.com/vektra/mockery/v2/pkg/config.SemVer=2.4.0
```

**标志说明**:
- `-s`: 去除符号表
- `-w`: 去除 DWARF 调试信息
- `-X`: 设置包变量值，这里设置 SemVer 为 2.4.0

设置正确的 SemVer 值很重要，因为 mockery 会在生成的代码顶部添加版本注释。如果不设置，版本会显示为 "v0.0.0-dev"，导致代码差异。

### 构建命令执行

脚本使用 `subprocess.check_output()` 执行外部命令，所有命令输出都会被打印，便于调试和日志记录。构建过程中的关键步骤：

1. **克隆仓库**: 使用完整克隆而非浅克隆
2. **版本检出**: 精确检出 v2.4.0 标签
3. **依赖更新**: 先执行 `go get` 更新特定依赖，再执行 `go mod tidy`
4. **二进制构建**: 使用 `go build` 配合自定义链接器标志

### 清理机制

构建完成后，脚本会：
1. 将二进制文件复制到目标目录外
2. 删除整个克隆的仓库目录树
3. 确保最终输出目录只包含单一的二进制文件

这种清理机制保证资产包的体积最小化。

## 设计模式与设计决策

### 函数式编程风格

脚本采用简单的函数式风格，而非面向对象设计。这种选择适合于：
- 一次性执行的任务脚本
- 线性的执行流程
- 最小化的状态管理

### 版本固定策略

脚本固定了 mockery 的版本（v2.4.0），而不是使用最新版本。这个决策基于：
- **可重现性**: 确保资产构建结果一致
- **兼容性**: 避免新版本引入的破坏性变更
- **稳定性**: 经过测试的版本更可靠

### 从源码构建

选择从源码构建而非下载预编译二进制，原因包括：
- **定制需求**: 需要应用特定的补丁和依赖更新
- **版本控制**: 精确控制所有依赖的版本
- **透明度**: 构建过程完全可见和可审计

### 环境隔离

脚本使用临时目录进行构建，然后复制产物并清理，避免污染目标环境。

## 性能考量

### 构建时间

构建 mockery 涉及多个步骤：
- Git 克隆: 网络 I/O 密集，取决于网络速度
- 依赖下载: `go get` 和 `go mod tidy` 需要下载依赖包
- 编译: CPU 密集，但 mockery 代码量不大，编译时间可控

### 磁盘空间

脚本使用临时空间存储：
- 完整的 Git 仓库
- Go 模块缓存
- 构建产物

最终输出只保留单一二进制文件，体积约几 MB。

### 优化策略

- **串行执行**: 构建步骤必须串行，无法并行化
- **清理及时**: 每步完成后立即清理不需要的文件
- **输出过滤**: 使用 `check_output()` 捕获但不显示所有输出，减少日志噪音
